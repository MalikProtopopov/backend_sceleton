# Sitemap и robots.txt: домен, протокол, локализация

Ответы на вопросы по текущей реализации бэкенда и что должно быть настроено для SEO.

---

## 1. Домен в URL (`<loc>`)

**Текущее поведение:** Базовый URL для sitemap берётся **из запроса** (`request.url.scheme` + `request.url.netloc`). То есть если sitemap запрашивают с `https://api.mediann.dev/...`, в тегах `<loc>` будет `https://api.mediann.dev/...`, а не домен фронта.

- **Откуда берётся базовый URL:** из `get_validated_base_url()` в `backend/app/modules/seo/router.py`: используется URL текущего запроса. Если домен запроса не входит в `allowed_domains` tenant, подставляется fallback: `tenant.domain` (без схемы) с той же схемой, что и запрос (`request.url.scheme`).
- **Настройка site_url / frontend_url:** в текущей модели **нет** поля `site_url` или `frontend_url`. Есть только:
  - `Tenant.domain` — один домен (например, `mediann.dev`);
  - `TenantSettings.allowed_domains` — список разрешённых доменов для **проверки** входящего запроса.

**Нужно на бэкенде:**  
Добавить в настройки tenant поле **`site_url`** (полный URL фронта, например `https://mediann.dev`) и использовать его при генерации sitemap и robots.txt для формирования всех `<loc>` и директивы `Sitemap:`.

---

## 2. Протокол и домен по умолчанию

**Текущее поведение:** Схема (http/https) всегда берётся из запроса. Если запрос пришёл по `http://`, в sitemap будет `http://...`.

- **Production и https:** в коде **нет** принудительной подстановки `https` для production.

**Нужно на бэкенде:**  
При использовании `site_url` хранить и отдавать его как есть (рекомендуется задавать сразу с `https://`). При fallback на `tenant.domain` в production можно принудительно использовать `https`.

---

## 3. Локализация URL

**Текущее поведение:**

- Sitemap отдаётся **по локали** через query-параметр: `?tenant_id=...&locale=ru` или `locale=en`. Для каждой локали свой набор URL (контент фильтруется по locale).
- **Формат путей в `<loc>`:** без префикса локали. Используются шаблоны из `SitemapAggregatorService.URL_PATTERNS`:
  - статьи: `/blog/{slug}`;
  - темы: `/blog/topic/{slug}`;
  - кейсы: `/portfolio/{slug}`;
  - услуги: `/services/{slug}`;
  - команда: `/team/{slug}`;
  - документы: `/documents/{slug}`.
  То есть в sitemap сейчас **нет** префикса вида `/ru/blog/...` или `/en/blog/...`.

**Уточнить с фронтом:**

- Какой формат URL использует фронт: с префиксом локали (`/ru/blog/...`) или без (`/blog/...`)?
- Нужны ли отдельные sitemap по локалям (`/sitemap-ru.xml`, `/sitemap-en.xml`) или один общий?  
  Сейчас бэкенд поддерживает: один общий sitemap с параметром `locale=...` и sitemap index по сегментам и локалям (`sitemap-{segment}-{locale}.xml`).
- Параметр `locale=ru` влияет только на **контент** (какие статьи/услуги/кейсы по языку), на **структуру пути** в бэкенде не влияет — префикс `/ru/` в путях не добавляется.

**Нужно на бэкенде:**  
Генерировать URL в том же формате, что и фронт. Если фронт использует `/ru/blog/...`, в настройках tenant или в коде нужно добавить опцию (например, префикс локали в путь) и использовать её при сборке `path` для sitemap.

---

## 4. Логика формирования URL

**Текущая логика:** URL собирается как `{base_url}{path}` в `build_sitemap_url(base_url, path)` (`app/core/url_utils.py`). `path` нормализуется (leading slash, lowercase, без лишних слэшей).  
То есть: **один canonical URL на запись** в sitemap; отдельные альтернативы по локалям (hreflang) в sitemap не формируются.

**Уточнить:**  
Нужны ли в sitemap теги `xhtml:link` с `hreflang` для мультиязычных страниц или достаточно одного canonical URL на локаль (текущее поведение)?

---

## 5. Пагинация / размер sitemap

**Текущее поведение:**

- Ограничения на **количество URL в одном sitemap** в коде **нет** (рекомендация Google — до 50 000 URL на файл).
- **Sitemap index** поддерживается: эндпоинт `GET /api/v1/public/sitemap-index.xml` отдаёт индекс, в котором перечислены сегментные sitemap по типам контента и локалям: `sitemap-{segment}-{locale}.xml` (например, `sitemap-articles-ru.xml`). Разбиения по размеру (по 50k URL) нет — разбиение идёт только по сегментам и локалям.

**Нужно на бэкенде (по желанию):**  
При росте числа URL в одном сегменте можно добавить разбиение одного сегмента на несколько файлов (например, sitemap-articles-ru-1.xml, …) с лимитом 50 000 URL на файл.

## 6. robots.txt

**Текущее поведение:**

- Эндпоинт есть: **`GET /api/v1/public/robots.txt`** (query: `tenant_id=...`).
- В директиве `Sitemap:` подставляется URL **текущего API**:  
  `Sitemap: {base_url}/api/v1/public/sitemap.xml?tenant_id={tenant_id}`.  
  То есть `base_url` снова берётся из запроса; если запрос к API (`api.mediann.dev`), в robots будет указан sitemap на домене API, а не фронта.

**Нужно на бэкенде:**  
При наличии у tenant настройки **`site_url`** использовать её для директивы Sitemap, например:  
`Sitemap: {site_url}/sitemap.xml`  
(если фронт отдаёт sitemap по этому пути) или проксирует запрос к API — тогда можно оставить текущий URL, но тогда в `Sitemap:` должен быть именно домен фронта, с которого поисковики получают sitemap. Итог: в robots.txt в директиве `Sitemap:` должен быть URL **фронта** (например, `https://mediann.dev/sitemap.xml`), а не API.

## Краткий чеклист для бэкенда

| Вопрос / задача | Текущее состояние | Целевое |
|-----------------|-------------------|--------|
| Откуда берётся базовый URL для sitemap? | Из URL запроса (и fallback на tenant.domain) | Из tenant `site_url`, если задан |
| В sitemap подставляется https? | Схема от запроса | Для production — да (через site_url с https или принудительно) |
| Формат URL: с локалью или без? | Без префикса: `/blog/...`, `/services/...` | По структуре фронта (при необходимости — опция префикса локали) |
| Hreflang / отдельные sitemap по локалям? | Отдельные по locale через query и sitemap index по segment+locale | По требованиям SEO (при необходимости — hreflang в sitemap) |
| Ограничения по размеру sitemap? | Нет лимита 50k; есть sitemap index по сегментам и локалям | При необходимости — разбиение по 50k URL |
| robots.txt указывает на домен фронта? | Нет, указывает на base_url запроса (часто API) | `Sitemap: {site_url}/sitemap.xml` или аналог |

---

## Главное

Чтобы sitemap и robots.txt были корректны для SEO:

1. В настройках tenant должен быть настроен **`site_url`** (домен фронта, например `https://mediann.dev`).
2. Этот URL должен использоваться во всех `<loc>` в sitemap и в директиве `Sitemap:` в robots.txt.
3. Формат путей (с или без префикса локали) должен совпадать со структурой URL на фронте.

После введения `site_url` и его использования в sitemap и robots текущая проблема с `http://api.mediann.dev/...` в `<loc>` будет устранена.
